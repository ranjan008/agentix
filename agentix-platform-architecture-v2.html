⚡ Zero-Idle by Design
Only the watchdog (~5 MB) runs continuously. All agent processes are cold-started on demand and terminated post-execution. No compute cost at rest.
🔒 Secure by Default
Every agent invocation passes through the RBAC gateway. No agent can execute a tool, read a secret, or call an API without explicit policy allowance.
🔌 Channel Agnostic
Any channel can trigger any agent. The normalized trigger envelope abstracts the source so agents are completely decoupled from the ingress mechanism.
📦 Skills-First
Agents gain capabilities via modular skills — installed from SkillHub or defined as custom enterprise packages. Skills bundle tools, prompts, and RBAC in one unit.
The Watchdog Process
The always-on heartbeat — tiny, stateless, fast. The only process that never sleeps.

Design Goals

The watchdog is the single always-running process. It is intentionally kept minimal — no business logic, no LLM calls, no heavy dependencies. Its sole purpose is to listen, authenticate, route, and spawn.

Target Memory Footprint: ~5-15 MB RSS
Achievable with Go, Rust, or minimal Node.js. No framework overhead.
Single Binary Deployment
Can be installed as a systemd service, launchd daemon, or Windows Service. Optional — can also run as a serverless function itself (AWS Lambda with SQS trigger).
Non-blocking I/O on all channels
Uses epoll/kqueue or async I/O to listen to hundreds of channels without spawning threads per channel.
Spawn modes: fork, container, lambda invoke
Agent spawn strategy is pluggable — fork a child process, docker run, kubectl apply a Job, or invoke a Lambda/Cloud Run function.
Graceful degradation
If an agent fails to spawn, the watchdog queues the trigger with backoff and alerts via configured channels.

Skills Engine
Modular, composable capability packages — inspired by OpenClaw's skills system, hardened with RBAC and enterprise isolation

What is a Skill?

A skill is a self-contained capability package that bundles instructions, tools, prompts, and RBAC permissions into a single installable unit. Skills are to Agentix what plugins are to an IDE — agents gain new abilities by installing skills, not by rewriting agent code.

📦 Built-in Skills
Ship with Agentix core: web-search, file-ops, email-composer, calendar-manager, code-executor, data-analyzer, report-generator
🌐 SkillHub (Registry)
Community + verified marketplace. Browse, install, and version-lock skills. Think npm/PyPI for agent capabilities. Verified publisher badges.
🛠 Custom Skills
Define enterprise-private skills in your repo. Git-versioned, CI-tested. Can wrap internal APIs, proprietary workflows, and domain-specific logic.
🔒 RBAC-Gated
Each skill declares required permissions. Agent can only activate a skill if its role grants those permissions. Skills cannot escalate beyond their declared scope.

Skill Definition (YAML)

# skills/zendesk-support/SKILL.yaml apiVersion: "agentix/v1" kind: "Skill" metadata: name: "zendesk-support" version: "1.3.0" author: "agentix-community" verified: true tags: ["support", "ticketing", "zendesk"] spec: description: "Create, update, and triage Zendesk tickets with smart routing" instructions: "./SKILL.md" # Markdown prompt tools_provided: - "zendesk.create_ticket" - "zendesk.update_ticket" - "zendesk.search_tickets" - "zendesk.assign_ticket" tools_required: - "http_request" # from platform secrets_required: - name: "ZENDESK_API_KEY" vault_path: "vault://zendesk/api_key" rbac: permissions_needed: - "zendesk:read" - "zendesk:write" data_scope: "tenant" config: zendesk_subdomain: "{{env.ZENDESK_SUBDOMAIN}}" auto_assign: true priority_mapping: urgent: "p0" high: "p1" normal: "p2"

Security & RBAC
Multi-layer security model — every call is authenticated, every action is authorized, everything is audited

Security Layers (Defense in Depth)

Layer 1 — Channel Authentication
Webhook HMAC signatures, Slack signing secrets, mTLS for gRPC. Unauthenticated requests never reach the watchdog core.
Layer 2 — Identity Verification (AuthN)
JWT / OAuth2 / API key validation against IdP (Keycloak, Auth0, AWS Cognito). Token claims carry role + tenant context.
Layer 3 — RBAC Authorization (AuthZ)
Policy engine (OPA / Casbin) evaluates: can this identity, in this role, trigger this agent, from this channel, on this tenant's data?
Layer 4 — Skill + Tool-Level Permissions
Each skill and tool call inside the agent runtime is validated. An agent cannot activate a skill or call a tool not in its allowed list.
Layer 5 — Data Scoping
Agents can only read/write data within their tenant/user scope. Row-level security enforced at the state store and vector DB layers.
Layer 6 — Immutable Audit Log
Every trigger, spawn, skill activation, tool call, output, and shutdown is written to an append-only audit log with cryptographic chaining.

RBAC Policy Example (OPA)

# policy.rego — Open Policy Agent package agentix.authz default allow = false allow { role := input.caller.roles[_] perm := data.roles[role].permissions[_] perm.agent == input.agent_id perm.action == "trigger" channel_allowed(perm, input.channel) } # Allow skill activation allow_skill { agent_def := data.agents[input.agent_id] input.skill_name == agent_def.skills[_] skill_perms_satisfied } # Allow tool call inside agent allow_tool { agent_def := data.agents[input.agent_id] input.tool_name == agent_def.tools[_] within_data_scope } within_data_scope { input.resource.tenant_id == input.caller.tenant_id }

Intelligent Scheduler
Three modes of scheduling: cron-based, event-based, and dependency-chain DAGs

Scheduling Modes

⏰ Cron
Standard cron expressions. Timezone-aware. Supports second-level precision.
📅 One-Shot
Fire once at a specific datetime. Used for reminders, delayed tasks, SLA deadlines.
🔗 DAG Chain
Agent A completes -> triggers Agent B -> fan-out to C and D. Full dependency graph.

Cron Schedule Examples

0 9 * * MON-FRI

0 0 1 * *

Multi-Agent Orchestration
Complex workflows — sequential chains, parallel fan-out, hierarchical delegation, and human-in-the-loop
🔀 Sequential Chain
Agent A -> Agent B -> Agent C. Each agent receives the output of the previous as context. Used for research -> analysis -> report pipelines.
  • Output passing via state store
  • Each agent has its own RBAC context
  • Failure in any step halts chain with rollback
🌿 Parallel Fan-Out
One trigger spawns N agents simultaneously. Results aggregated by a collector agent. Ideal for multi-source research.
  • Configurable concurrency limit
  • Partial result handling
  • Timeout-based aggregation
🎯 Supervisor Pattern
A planner/orchestrator agent dynamically decides which specialist agents to call based on LLM reasoning.
  • Dynamic routing at runtime
  • Supervisor has elevated RBAC
  • Max delegation depth enforced
🤝 Human-in-the-Loop
Agent pauses at checkpoints, requests human approval via any channel, then resumes or aborts.
  • Approval timeout with auto-abort
  • Approval link with JWT expiry
  • Audit trail of human decisions
🔄 Retry + Compensation
Failed agents trigger compensation actions (saga pattern). Exponential backoff with jitter. Dead-letter queue.
  • Configurable backoff strategy
  • Compensating agent definition
  • Idempotency key enforcement
🔁 Event-Driven Cascade
Agent emits domain events on completion. Other agents subscribed to those events automatically trigger. Fully decoupled.
  • Event schema registry
  • At-least-once delivery
  • Event replay for debugging
Memory & State
Agents are ephemeral but their memory is not — three-tier memory architecture with tiered infrastructure options
⚡ Tier 1: Working Memory
TTL-based, in-memory store for a single agent execution or conversation session.
  • Lite: SQLite in-memory / file
  • Standard: Redis / Valkey
  • Conversation history (last N turns)
  • Intermediate tool results
  • TTL: configurable (default 1hr)
🧠 Tier 2: Semantic Memory
Embeddings-based retrieval of past interactions and knowledge.
  • Lite: sqlite-vec (zero-dep)
  • Standard: pgvector / Chroma
  • Auto-embed agent outputs
  • Similarity search at context-load
  • Configurable retention policy
🗄 Tier 3: Structured Memory
Relational storage for structured knowledge and entities.
  • Lite: SQLite (single file)
  • Standard: PostgreSQL
  • Named memory slots (key-value)
  • Entity store (people, orgs)
  • Encrypted at rest
Tool Registry
Centrally managed catalog of all tools available to agents, with per-tool RBAC and rate limiting
🔌 Built-in Tools
web_search, web_fetch, file_read, file_write, bash_exec, http_request, email_send, calendar_*, database_query
🔗 MCP Connectors
Slack, GitHub, Linear, Notion, Jira, Salesforce, Zendesk, HubSpot, GDrive — any MCP-compatible service
⚙ Custom Tools
Register any function as a tool via OpenAPI spec or simple function decorator. Auto-generates JSON schema.
🤖 Agent-as-Tool
Any agent can be called as a tool by another agent. Enables recursive, composable agent architectures.

Key Differentiators vs. OpenClaw

Serverless vs. Always-On
OpenClaw runs an always-on Node.js gateway process (~200-500 MB RAM). Agentix's watchdog is ~5 MB, and agents scale to zero. OpenClaw's gateway must be running 24/7 even when idle.
Enterprise Security vs. User-Managed
OpenClaw's base platform has no RBAC or formal compliance — it relies on the user to sandbox properly. Enterprise RBAC requires the paid Adopt AI fork. Agentix bakes 6-layer security into the core.
Skills: Inspired but Hardened
Agentix adopts OpenClaw's excellent SKILL.md pattern but adds RBAC-gated activation, permission declarations, secrets management, and a verified publisher system. Skills can't escalate privileges.

The Bottom Line on Cost

Agentix Lite is the cheapest option
With SQLite + sqlite-vec + in-process cache, Agentix Lite needs zero external services. A $5/mo VPS (or even a Raspberry Pi) can run the entire stack. The LLM API is always the real cost — not infrastructure.
Standard/Enterprise adds cost for a reason
PostgreSQL + Redis are only needed for multi-user production, HA, and compliance. Yes, that adds ~$25-50/mo in managed services — but you get concurrent access, ACID transactions, proper vector search, and enterprise audit. OpenClaw requires similar infra at the same scale.
